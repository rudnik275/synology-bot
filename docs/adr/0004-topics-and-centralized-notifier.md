# Centralized OwnerNotifier; per-category private-chat forum topics

В Bot API 9.4 (февраль 2026) Telegram разрешил `createForumTopic` в **приватных** чатах. До этого все push-уведомления бота (готовое скачивание, stuck-task, NAS down, disk full, Watchtower deploy, дайджест подписок) сыпались в один плоский DM — owner не мог визуально разделить «надо посмотреть сейчас» от «можно завтра». Решение: создать четыре именованных топика (`Торренты`, `Состояние NAS`, `Деплой`, `Подписки`) при первом запуске и роутить каждый тип уведомления в свой тред. Одновременно — централизовать само отправление: до этого в `app.ts` лежало 5 копий паттерна `if (!owner_chat_id) return; bot.api.sendMessage(...)`, каждый watcher знал про `chat_id` напрямую. Вместо этого `OwnerNotifier.send(category, text, opts?)` — единственная точка, где живёт логика «куда уходит сообщение».

Альтернативы: (1) оставить плоский чат и красить сообщения emoji-префиксами — owner просил топики явно, emoji не дают визуальной группировки и не сворачиваются; (2) роутить каждый watcher отдельно, передавая `threadId` в его опции — дублирование «достать chat_id, упасть если нет» сохраняется, при добавлении категории нужно править 5+ файлов. **Watchtower-уведомления** идут отдельным путём: shoutrrr из docker-compose убран, бот сам при старте сравнивает ImageID своего контейнера с прошлым (KV `bot_image_sha`) — это даёт ровно одно сообщение на реальный деплой и не зависит от формата watchtower-логов (`WATCHTOWER_NOTIFICATION_REPORT`, log-level, etc.). Failed deploys, где бот не стартует, естественно не нотифицируются — owner замечает по тишине и зовёт `/deploy-status`. Fallback: если `createForumTopic` вернул 4xx (старый клиент / `allows_users_to_create_topics=false`), KV-флаг `topics_disabled` ставится один раз, дальше `OwnerNotifier` шлёт без `message_thread_id` — owner получает всё в плоский DM как раньше, без поломки.
