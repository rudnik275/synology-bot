# cloudflared в namespace бота + Watchtower depends-on (туннель переживает деплои)

[[0005]] решило выставлять Mini App наружу через Cloudflare Tunnel на самом NAS (loopback-порт, ноль открытых портов). При go-live (релизы v3.4.0–v3.4.3) уточнили **как** туннель уживается с ботом в рантайме. cloudflared — отдельный контейнер в том же compose-стеке, который **делит сетевой namespace бота** (`network_mode: "service:bot"`), а не сидит на общей user-сети. Причина: Hono-сервер биндится на `127.0.0.1:${MINIAPP_PORT}` (loopback — не выставлен ни на LAN, ни на хост, ровно как требует ноль-портов из [[0005]] и owner-only из [[0001]]); чтобы отдельный контейнер дотянулся до этого loopback, ему нужен тот же netns. Альтернатива — общая docker-сеть + bind `0.0.0.0` + маршрут на `http://synology-bot:8080` — отвергнута: `0.0.0.0` шире, чем нужно для single-user-сервиса, и публичный hostname в Cloudflare пришлось бы вешать на имя сервиса вместо `localhost`.

Цена netns-шеринга: пересоздание контейнера бота (а это **каждый** Watchtower-апдейт образа) рвёт netns у cloudflared → туннель уходит в Down, Mini App недоступен до ручного рестарта cloudflared. Наступали на это на каждом релизе. Лечится **меткой Watchtower** на cloudflared: `com.centurylinklabs.watchtower.depends-on=synology-bot` (+ `enable=true`) — Watchtower пересоздаёт cloudflared сразу после апдейта бота, и туннель сам поднимается на каждом деплое. Без метки альтернативой был бы внешний healthcheck-autoheal или отказ от netns-шеринга — обе тяжелее одной строки в compose.

Два операционных факта, на которых тоже споткнулись и которые фиксируем, чтобы не re-litigate: **туннель и зона `rudifamily.uk` обязаны быть в одном Cloudflare-аккаунте** — Public Hostname создаёт DNS-запись `nas → <tunnel-id>.cfargotunnel.com` только в пределах учётки зоны (туннель, сделанный в чужом аккаунте, маршрут не создаёт). И **Service URL хостнейма — `http://localhost:8080`** (именно HTTP: Hono за туннелем — простой HTTP, не TLS). Релиз — ручной пуш SemVer-тега `v3.4.x` после зелёной чистой сборки (тег = первая чистая Linux-сборка фронта, см. практику из [[0006]]); Watchtower тянет `:latest` каждые 300с. Полная топология прода — в memory `project_deployment`.
